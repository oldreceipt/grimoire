Source 2 preview - blend-state testbed

Standalone. Does not touch the live 3D preview. Each panel renders the current grimoire behavior (left) next to the corrected behavior (right) so the difference is visible. three.js 0.184.0 (matches grimoire's pin).

Mirrors the blend rules in src/lib/source2Preview/drawState.ts and src/lib/deadlockMaterial.ts. Read-only reproduction; no app code imported.

1. Additive blend is alpha-weighted, not pure additive

drawState.ts:62 & deadlockMaterial.ts:327 set THREE.AdditiveBlending

AdditiveBlending (SrcAlpha · src + One · dst) CustomBlending (One · src + One · dst)

THREE.AdditiveBlending is blendSrc = SrcAlphaFactor, so the framebuffer gets src.rgb * src.a + dst.rgb. Where the glow texture's alpha falls below 1 (its soft halo), the added light is attenuated by that alpha. A true Source 2 additive pass adds the self-illum result regardless of geometry alpha (blendSrc = OneFactor). Same texture, same scale, same backdrop on both sides: the left halo is visibly tighter/dimmer.

Only identical if the overlay's alpha is 1 everywhere. Not forced to One.

2. No intra-mesh transparency sorting

three.js sorts transparent objects by distance, never triangles within one mesh

one mesh (buffer order) separate meshes (depth-sorted)

Three overlapping translucent quads at different depths (R/G/B, opacity 0.5, depthWrite=false). Left = one merged mesh: three draws its triangles in buffer order, so the color stacking is wrong and flips as it rotates. Right = three separate meshes: three re-sorts them back-to-front every frame, so the stacking is correct. A self-overlapping translucent morphic mesh hits the left case.

Inherent three.js limitation; the draw-state pass does nothing to mitigate it.

3. renderOrder is per-mesh, not per-submaterial

drawState.ts:99-100,149-151 - "strongest overlay order among a multi-material mesh wins"

far-layer shares a mesh with the additive overlay layers are separate meshes

A mesh has two groups: a far translucent layer (cyan) and an additive overlay that must draw last, so the whole mesh's renderOrder is raised. That also drags the cyan layer past the separate near translucent layer (magenta), so cyan blends over magenta - wrong. Left = shared mesh (cyan wrongly on top); right = three separate meshes ordered far → near → additive (magenta correctly on top). A submaterial cannot carry its own draw order.

Bites only on a mesh that mixes an overlay submaterial with ordered transparency.
